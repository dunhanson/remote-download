# 文件离线下载服务 - 实现计划

## Context

当前项目仅有文档（需求说明 + UI 预览），无实际代码。需要从头构建一个基于 Nuxt 4 的文件离线下载服务，支持：用户提交 URL、异步下载、进度监控、完成后生成下载链接。

## 技术架构

| 层级 | 技术 |
|------|------|
| 框架 | Nuxt 4 (全栈) |
| UI | Element Plus + 自定义组件 |
| 数据库 | SQLite (better-sqlite3) |
| 队列 | Redis + Bull |
| 文件服务 | 本地磁盘（Nuxt server API 代理） |
| 下载 | axios stream + Range 支持 |
| 认证 | 请求头 `X-User-Id` + localStorage（前端缓存用户ID） |

## 下载签名认证设计

### 用户下载密钥

每个用户拥有一个下载密钥 `download_key`，存储在 users 表中。

```sql
-- users 表新增字段
ALTER TABLE users ADD COLUMN download_key TEXT;  -- 用户下载密钥
```

### 签名生成算法

```typescript
interface DownloadSignature {
  expiresAt: number     // 过期时间戳
  signature: string     // MD5 签名
}

// 生成下载签名
function generateDownloadSignature(userId: string, userDownloadKey: string, expiresInSeconds: number = 3600): DownloadSignature {
  const expiresAt = Math.floor(Date.now() / 1000) + expiresInSeconds

  // 签名字符串: {userId}&{userDownloadKey}&{expiresAt}
  const stringToSign = `${userId}&${userDownloadKey}&${expiresAt}`
  const signature = createHash('md5').update(stringToSign).digest('hex')

  return { expiresAt, signature }
}

// 验证签名
function verifySignature(
  userId: string,
  params: { signature: string; expires: string }
): boolean {
  const { signature, expires } = params
  const expiresAt = parseInt(expires)

  // 检查是否过期
  if (Math.floor(Date.now() / 1000) > expiresAt) {
    return false
  }

  // 从数据库获取用户的 download_key
  const userDownloadKey = getUserDownloadKey(userId)
  if (!userDownloadKey) {
    return false
  }

  // 重新计算签名并比对
  const stringToSign = `${userId}&${userDownloadKey}&${expires}`
  const expectedSignature = createHash('md5').update(stringToSign).digest('hex')

  return signature === expectedSignature
}
```

### 签名参数说明

| 参数 | 说明 |
|------|------|
| `userId` | 用户 ID，从请求头 X-User-Id 获取 |
| `expires` | 过期时间（Unix时间戳），默认1小时后过期 |
| `signature` | MD5 签名 = `MD5(filename + "&" + userDownloadKey + "&" + expiresAt)` |

## 认证设计

### 认证流程

```
前端首次访问:
  1. 检查 localStorage 中是否有 userId
  2. 没有 → 调用 GET /api/user/create 生成新用户
  3. 服务端生成 MD5(UUID) 作为 userId，创建 users 记录
  4. 前端将 userId 存入 localStorage

前端后续请求:
  1. 从 localStorage 获取 userId
  2. 在请求头添加 X-User-Id: {userId}
  3. 服务端验证用户存在且状态正常

无用户态请求（如文件下载）:
  1. URL 中携带 userId、expires、signature 参数
  2. 服务端验证签名有效性
```

### 用户标识生成规则

```typescript
// 使用 MD5(UUID) 作为用户标识
import { v4 as uuidv4 } from 'uuid'
import { createHash } from 'crypto'

function generateUserId(): string {
  return createHash('md5').update(uuidv4()).digest('hex')
}
// 示例: a1b2c3d4e5f6a1b2c3d4e5f6a1b2c3d4
```

### 前端存储

```typescript
// 存储用户 ID
localStorage.setItem('userId', 'a1b2c3d4e5f6a1b2c3d4e5f6a1b2c3d4')

// 请求时带上
headers: {
  'X-User-Id': localStorage.getItem('userId')
}
```

## 数据库模型

### users 表 - 用户表

```sql
-- users 表 - 用户记录
CREATE TABLE users (
    id TEXT PRIMARY KEY,                    -- 用户唯一ID（MD5(UUID)）
    download_key TEXT NOT NULL,             -- 用户下载密钥（用于生成下载链接签名）
    ip_address TEXT,                        -- 最后访问 IP 地址
    user_agent TEXT,                         -- 用户浏览器 UA
    created_at INTEGER NOT NULL,             -- 创建时间（Unix时间戳）
    last_active_at INTEGER NOT NULL,         -- 最后活跃时间（Unix时间戳）
    login_count INTEGER DEFAULT 1,           -- 登录次数
    status TEXT DEFAULT 'active'             -- 状态：active(正常)/banned(封禁)
);

CREATE INDEX idx_users_created_at ON users(created_at DESC);  -- 按创建时间索引
```

### login_logs 表 - 用户登录日志表

```sql
-- login_logs 表 - 用户登录日志
CREATE TABLE login_logs (
    id INTEGER PRIMARY KEY AUTOINCREMENT,     -- 日志自增ID
    user_id TEXT NOT NULL,                  -- 用户ID（关联 users 表）
    ip_address TEXT NOT NULL,               -- 登录 IP 地址
    user_agent TEXT,                         -- 登录时浏览器 UA
    created_at INTEGER NOT NULL,             -- 登录时间（Unix时间戳）
    event_type TEXT DEFAULT 'login',         -- 事件类型：login(登录)/logout(登出)/create(创建用户)
    FOREIGN KEY (user_id) REFERENCES users(id)
);

CREATE INDEX idx_login_logs_user_id ON login_logs(user_id);                      -- 按用户ID查询索引
CREATE INDEX idx_login_logs_created_at ON login_logs(created_at DESC);            -- 按登录时间倒序索引
```

### tasks 表 - 下载任务表

```sql
-- tasks 表 - 下载任务记录
CREATE TABLE tasks (
    id TEXT PRIMARY KEY,                    -- 任务唯一ID (UUID)
    user_id TEXT NOT NULL,                  -- 用户ID（MD5(UUID)，关联 users 表）
    source_url TEXT NOT NULL,               -- 原始下载地址
    filename TEXT NOT NULL,                 -- 保存的文件名
    filesize INTEGER DEFAULT 0,              -- 文件大小（字节），0表示未知
    downloaded INTEGER DEFAULT 0,            -- 已下载字节数
    status TEXT DEFAULT 'pending',           -- 状态：pending(等待中)/downloading(下载中)/completed(已完成)/failed(失败)
    speed INTEGER DEFAULT 0,                 -- 当前下载速度（字节/秒）
    error_message TEXT,                      -- 失败时的错误信息
    retry_count INTEGER DEFAULT 0,           -- 当前重试次数
    max_retries INTEGER DEFAULT 3,           -- 最大重试次数
    local_path TEXT,                        -- 本地存储相对路径（相对于 rootPath）
    created_at INTEGER NOT NULL,             -- 创建时间（Unix时间戳）
    updated_at INTEGER NOT NULL,             -- 最后更新时间（Unix时间戳）
    completed_at INTEGER                     -- 完成时间（Unix时间戳），null表示未完成
);

CREATE INDEX idx_tasks_user_id ON tasks(user_id);                          -- 按用户ID查询索引
CREATE INDEX idx_tasks_status ON tasks(status);                            -- 按状态查询索引
CREATE INDEX idx_tasks_created_at ON tasks(created_at DESC);                -- 按创建时间倒序索引
```

### 表关系

```
users (1) ──────< (N) tasks
  │
  └─ id (PK) ──────────> user_id (FK)
```

## API 设计

### 认证相关

| 方法 | 路径 | 说明 |
|------|------|------|
| GET | `/api/user/create` | 创建新用户（首次访问时调用） |
| GET | `/api/user/info` | 获取当前用户信息（需 X-User-Id 请求头） |

**说明**: 任务相关 API 需要 `X-User-Id` 请求头识别用户身份。

### 任务相关

| 方法 | 路径 | 说明 |
|------|------|------|
| POST | `/api/tasks` | 批量创建任务 |
| GET | `/api/tasks` | 任务列表（状态过滤，最多返回10个，按创建时间倒序） |
| GET | `/api/tasks/:id` | 任务详情（仅属于当前用户） |
| DELETE | `/api/tasks/:id` | 删除任务（仅属于当前用户） |
| POST | `/api/tasks/:id/retry` | 重试失败任务（仅属于当前用户） |
| GET | `/api/tasks/:id/progress` | 实时进度 (SSE) |

### 文件相关

| 方法 | 路径 | 说明 |
|------|------|------|
| GET | `/files/:filename` | 通过签名下载文件（服务端代理） |

**下载链接格式**:
```
GET /files/{filename}?expires={expiresAt}&signature={signature}
```

| 参数 | 说明 |
|------|------|
| `filename` | 文件名（如 go1.22.0.darwin-arm64.tar.gz） |
| `expires` | 过期时间（Unix时间戳），超过此时间链接失效 |
| `signature` | MD5 签名 = `MD5(filename + "&" + userDownloadKey + "&" + expiresAt)` |

**说明**:
- 文件名需全局唯一，可在原文件名基础上拼接 taskId 前缀（如 `{taskId}_{filename}`）
- 服务端通过 filename 查找对应任务记录
- 验证签名时从任务记录中获取 userId，再查找用户的 download_key

## 核心实现步骤

### Phase 1: 项目初始化
1. `npx nuxi@latest init` 初始化 Nuxt 4 项目
2. 安装依赖: element-plus, better-sqlite3, ioredis, bull, uuid, axios
3. 配置 nuxt.config.ts 和 app.config.ts

### Phase 2: 配置层
- `config/download.ts` - 下载相关配置（路径、URL）

### Phase 3: 数据库层
- `server/database/schema.ts` - 建表语句（users + tasks + login_logs）
- `server/database/index.ts` - 连接管理
- `server/database/users.ts` - 用户 CRUD 操作
- `server/database/tasks.ts` - 任务 CRUD 操作

### Phase 4: 用户 API
- GET /api/user/create - 创建新用户
- GET /api/user/info - 获取用户信息（验证 X-User-Id 请求头）

### Phase 5: 基础任务 API
- POST /api/tasks - 创建任务并入队
- GET /api/tasks - 列表查询（按 X-User-Id 请求头过滤）
- GET /api/tasks/:id - 详情
- DELETE /api/tasks/:id - 删除

### Phase 6: 任务队列 + Worker
- `server/services/queue.ts` - Bull 配置
- `server/workers/download.worker.ts`:
  - HEAD 请求获取文件大小
  - 分段 stream 下载
  - 进度更新 (throttled 500ms)
  - 断点续传支持
  - 失败重试 (exponential backoff)

### Phase 7: 文件下载代理 + 签名验证
- GET /files/:filename - 通过签名验证下载文件

### Phase 8: 前端页面
- 参考 `docs/preview-v4.html` 实现 UI
- `pages/index.vue` - 整合所有组件
- `composables/useUser.ts` - 用户状态管理（localStorage 存储 userId）
- `composables/useTasks.ts` - 任务状态管理 + 轮询

### Phase 9: 实时进度
- GET /api/tasks/:id/progress - SSE 接口
- 前端每 2-3 秒轮询更新

## 验证方式

1. 启动 Nuxt dev server (`npm run dev`)
2. 首次访问前端，检查 localStorage 中是否生成 `userId`
3. 调用 GET /api/user/create 手动测试用户创建
4. 检查数据库 users 表和 login_logs 表是否有新记录
5. 提交一个测试 URL（如 GitHub release 文件）
6. 观察任务状态变化（pending → downloading → completed）
7. 检查文件是否正确保存到配置的 storage/downloads/ 目录
8. 测试重试、删除、主题切换等功能
9. 测试文件下载链接签名是否正确生成并可访问
10. 清除 localStorage，验证新用户创建逻辑
